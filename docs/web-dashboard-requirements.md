# コルディオグループ Web ダッシュボード 要件定義書

最終更新: 2026-06-15

## 1. 目的

既存の `minpakuIN-download/create_report.py` と `コルディオレポートNEW.xlsm` で生成している施設別レポートを、Vercel + Supabase + Next.js の Web ダッシュボードとして再構築する。

今後は minpakuIN だけでなく、ねっぱん、手間いらずの予約データを同じダッシュボードで扱えるようにする。そのため、PMS/OTA 管理システムごとの生データを直接画面表示に使わず、専用データテンプレートへマッピング・変換してから集計する。

## 2. 背景と現状

### 2.1 現行構成

現行は minpakuIN の `base.csv` を `create_report.py` で集計し、`集計データ.xlsx` を作成している。`コルディオレポートNEW.xlsm` は、その集計データを参照し、以下の分析シートを表示している。

| 分析 | 現行シート |
| --- | --- |
| 稼働分析 | `稼働分析表(月間)`, `稼働分析表(年間)` |
| 経路分析 | `経路別分析表(月間)`, `①経路別分析表(年間)`, `②経路別分析表(年間)` |
| 国籍別分析 | `国籍別分析(NEW)` |
| 泊数分布 | `泊数分布(NEW)(部屋タイプ別)`, 集計元 `泊数分布` |
| 部屋タイプ別分析 | `部屋タイプ別分析(NEW) (全施設)` |
| 全施設年間売上 | `全施設年間売上` |
| ブッキングカーブ | `ブッキングカーブ`, 集計元 `ブッキングカーブ集計` |

### 2.2 現行集計の前提

`create_report.py` で確認した主なルールは以下。

| 項目 | 現行ルール |
| --- | --- |
| 施設名 | `アクアパレス北谷` の一部部屋タイプを別施設へ分割。`琉心 恩納` は表示名へ変換 |
| 部屋利用月 | `部屋利用日` の月初日 |
| 親子判別 | `部屋利用日 == チェックアウト日` はチェックアウト行として除外対象 |
| 室数 | 現行 minpakuIN では `親子判別=1` の行数 |
| キャンセル | `ステータス == "キャンセル済み"` を通常集計から除外 |
| Agoda 補正 | 2026-01-01 以降の Agoda は `宿泊費 / 0.88` |
| Trip.com 補正 | 2026-02-01 以降の Trip.com 系は `宿泊費 / 0.85` |
| 税抜 | `宿泊費 - 消費税` |
| 泊数分布 | `施設名 + OTA予約番号 + 部屋タイプ` を予約単位として集計 |
| ブッキングカーブ | `部屋利用日 - 予約受付日` のリードタイムを累積室数化 |

## 3. スコープ

### 3.1 対象

- ねっぱん、手間いらず、minpakuIN のデータを共通テンプレートに変換する取り込みミニアプリ
- Supabase への raw データ保管、正規化データ保管、集計マート作成

### 3.2 対象外

- 手間いらずのデイリーダウンロードアプリ本体の詳細実装
- ねっぱん/手間いらず側の管理画面操作自動化の詳細実装
- Claude Design に委任する最終ビジュアルデザイン

## 6. 共通データテンプレート要件

各ソースは最終的に以下のテンプレートへ正規化する。

### 6.1 正規化粒度

原則として `1レコード = 1施設 × 1予約 × 1部屋タイプ × 1宿泊日` とする。

ただし、手間いらずやねっぱんのように `部屋数` / `室数` を持つデータは、物理的に複数行へ展開せず、`sold_room_nights` に室泊数を保持してもよい。集計では `COUNT(*)` ではなく `SUM(sold_room_nights)` を室数として使う。minpakuIN 由来データでは `sold_room_nights=1` として現行集計と一致させる。

### 6.2 必須項目

| 区分 | 項目 |
| --- | --- |
| ソース識別 | `source_system`, `source_facility_code`, `source_file_id`, `ingest_batch_id` |
| 施設 | `facility_id`, `facility_name`, `area_name` |
| 予約 | `reservation_key`, `checkin_code`, `ota_reservation_no`, `status`, `channel` |
| 日付 | `stay_date`, `stay_month`, `checkin_date`, `checkout_date`, `booked_at` |
| 部屋 | `room_type_raw`, `room_type_normalized`, `budget_room_type`, `room_no` |
| 人数 | `guest_count`, `adult_count`, `child_count` |
| 泊数 | `nights`, `stay_night_index`, `sold_room_nights` |
| 金額 | `gross_amount`, `tax_amount`, `net_amount`, `fee_adjusted_gross_amount`, `fee_adjusted_tax_amount`, `fee_adjusted_net_amount`, `fee_adjustment_rule_id` |
| 国籍 | `country_raw`, `country_normalized`, `country_major`, `country_middle` |
| 集計制御 | `is_stay_night`, `is_cancelled`, `is_valid_lead_time`, `lead_time_days` |

KPI の正式な計算式は `docs/kpi-definitions.md` を正とする。

## 9. データレイク/DB 要件

### 9.1 推奨構成

初期構成は Supabase Storage + Supabase Postgres を採用する。

| レイヤー | 保存先 | 内容 |
| --- | --- | --- |
| Raw lake | Supabase Storage private bucket | ねっぱん/手間いらず/minpakuIN の原本 CSV/XLSX |
| Ingest metadata | Supabase Postgres | 取込バッチ、ファイル、検証結果、マッピング設定 |
| Canonical data | Supabase Postgres | 共通テンプレートへ正規化した宿泊日別データ |
| Mart | Supabase Postgres | ダッシュボード用の月次/日次集計テーブルまたは materialized view |

理由:

- raw ファイルを保管し、後からマッピング修正・再集計できる
- Supabase Auth/RLS と施設別権限を同じ Postgres 上で扱える
- Vercel Function で重い集計を毎回実行せず、事前集計マートを API から読む構成にできる

### 9.2 将来拡張条件

以下のいずれかを満たした場合は、raw/正規化データを Parquet 化して S3 互換ストレージ、BigQuery、DuckDB/Trino 系の lakehouse へ分離する。

- `reservation_stay_nights` が 3,000 万行を超える
- 月次集計の再計算が 10 分を超える
- ダッシュボード API の P95 が 3 秒を超える
- 複数会社/複数ブランド横断の分析が増える

## 10. 集計最適化要件

Web 版では、画面表示時に raw データから都度 group by しない。以下の流れにする。

1. raw ファイルを Storage に保存
2. staging にロード
3. canonical に upsert
4. mart を差分更新
5. API は mart を読む

更新対象は取込バッチ内の `stay_month` と影響施設に限定する。全期間再集計は手動メンテナンス操作に限定する。

## 11. セキュリティ要件

- Supabase Auth を利用する
- Supabase RLS で施設別閲覧権限を強制する
- service role key はサーバー/バッチ処理だけで使い、ブラウザへ出さない
- raw ファイル bucket は private にする
- パスワード、token、refresh token はログ・画面・成果物に出さない
- `.env.local` は上書き・削除しない

## 12. 検証要件

検証基準は `コルディオレポートNEW.xlsm` との数値一致とする。

| 対象 | 検証方法 | 許容差 |
| --- | --- | --- |
| 室数/予約件数 | Excel の該当シートと API/mart を突合 | 完全一致 |
| 金額 | 税込/税抜、施設、月、経路、部屋タイプ別で突合 | 1円以内 |
| 比率 | 稼働率、構成比、連泊率など | 0.01pt 以内 |
| ADR/RevPAR | Excel 表示丸め後と突合 | 1円以内 |
| ブッキングカーブ | リードタイム区分別に突合 | 完全一致 |

設計確認では `コルディオレポートNEW.xlsm` を読み取り、39シート構成と対象ダッシュボードシートを確認した。
