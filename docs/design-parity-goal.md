# 宿泊BIダッシュボード — Claude Design 視覚パリティ Goal

ステータス: 策定（2026-06-22）/ ブランチ `feat/backend-foundation`
位置づけ: `docs/implementation-plan.md`（全体計画 M0-M22）のうち **「フロント視覚パリティ」を独立ゴールとして切り出したもの**。バックエンド/データ層は実装・±0検証済み（[[backend-complete-state]]）。

## 0. 目的（Goal）

実装中の Next.js ダッシュボードを、Claude Design プロトタイプ **「宿泊BIダッシュボード」**（`docs/webdashboard-app2/project/`）に **視覚的・構造的に一致**させる。

> **最優先＝各画面の「中核テーブル／マトリクスの表示形式」をデザイン通りに再現する。**
> 同一フィルタ状態でデザインのスクリーンショットと並べたとき「同じ構造の画面に見える」状態をゴールとする。

きっかけ: 稼働分析・経路分析・全施設年間売上を実画面で確認したところ、デザインの中核である密なクロスタブ表が、フラット表／ドーナツ／単一表＋チャートに置き換わっており「テーブルの表示の仕方が全然違う」状態だった（2026-06-22 ユーザー指摘 + 実測スクショで確認）。

## 1. 現状診断（実測 + 10エージェント・コード監査）

**全体パリティ ≈ 70%（部分的に忠実）。** トークン・共通UI部品・チャート基盤は忠実移植済み。差異の主因は次の3点。

1. **中核クロスタブ表の消失（4画面）** — デザインの主役「行×12ヶ月／×施設の密なExcel風クロスタブ」が、ライブAPIに月次/施設別グリッドが無いためフラット表・指標ピボット・ドーナツに置換。**最大の体感差。**
2. **デザインに無い追加要素** — KPI strip＋ドーナツが複数画面に足され、本来は表だけだった画面のシルエットが変わり、表が折り返し以下に押し下げられている。
3. **occupancy の既定状態・欠落ビュー / shell のモバイル・UserMenu未実装。**

## 2. スコープ方針（前提と決定）

- **デザイン準拠を最優先。** デザインに無い追加要素（KPI strip／ドーナツ）は、デザインのシルエットを崩す画面では **削除またはサマリー帯へ縮小**し、主役の表を最上部に戻す。
- **既に忠実な部分は維持**（トークン `globals.css`・共通部品 `components/ui/*`・`components/charts/*`・stay-nights・dashboard）。再実装しない。
- **データ差は仕様通り維持＝ゴール対象外**: 実37〜41施設・サーバ計算（mart の `fee_adjusted_*`/`comparison`）・実データ無い箇所の「—」表示・税込/税抜のサーバ正。これらは“別物に見える”一因だが**意図的で正しい**。
- **RBAC/認証・運用系9ルート（取込/差分検証/マスタ/権限/設定）は別マイルストーン**（本ゴール外）。ただし shell の視覚要素（UserMenu 等）はスタブで先行してよい。

### 決定事項（デフォルト＝デザイン準拠。変更したい点があれば指示ください）

| 論点 | 決定（デフォルト） |
|---|---|
| 中核クロスタブ表 | **復元する**（mart/API契約を拡張し、デザイン通りの密なクロスタブを再構築） |
| 追加KPI/ドーナツ | **デザイン準拠で整理**（主役表を優先。KPIはサマリー帯/削除、ドーナツは原則撤去） |
| フィットレイアウト `calc(100dvh-152px)` | **回帰する**（1画面に収まるノンスクロール、マトリクスは内部スクロール） |
| occupancy 全施設ビュー＋比較モード | **本ゴールに含める** |
| モバイル/レスポンシブシェル | 含める（Phase 4、後ろ倒し可） |
| UserMenu / 鮮度・検証バッジ | スタブで先行（Phase 4） |

## 3. 成功基準（Done の定義）

- [ ] 各 analytics 画面を**デザインと同一フィルタ状態**にして並べたとき、中核テーブル/マトリクスの**構造が一致**して見える。
- [ ] channels / nationalities / room-types / annual-sales の **行×月（または×施設）クロスタブ**が復元されている（多段スティッキーヘッダー・エリア/年colSpan・合計行＋合計列・ヒートシェーディング）。
- [ ] occupancy が、全施設選択時に **AllFacilitiesビュー**を表示し、**比較モードセレクタ**（前年実績/前年同期/指定日付/予算差）が動く。デザインに無い**トレンドchartは撤去**（または明示的に残す判断）。
- [ ] デザインに無いKPI/ドーナツが**主役テーブルを押し下げていない**。
- [ ] analytics 各画面が **1画面フィット**（ノンスクロール）に回帰。
- [ ] shell に UserMenu・DataFreshnessBadge・ValidationBadge が表示され、**モバイル**でドロワー＋ボトムシートが動く。
- [ ] `next build` / `tsc` / `lint` クリーンを維持。実測スクショでデザイン一致を確認。

## 4. ロードマップ（優先順）

### Phase 1 — 即効・FEのみ・低工数/高インパクト
- occupancy: 既定比較を見直し（**3列マトリクスを既定/常時表示**化、または比較=前年実績を既定に）、デザインに無い**稼働トレンドchartを撤去**。
- dashboard: ヘッダーに「詳細分析へ」(Maximize2→occupancy)ボタンを追加。
- channels: エクスポートボタン追加、メタ行に「全N施設横断」スコープ表記を復元。
- occupancy/annual-sales/stay-nights: 「確定データ未取込／直近実績表示中」警告バナーを実装（対象年≠最新時）。
- occupancy: タイトル行インサイトを「予算まで残り・達成率・前年比」(Targetアイコン)へ。
- shell: FacilitySelector 各行に室数（右寄せ）表示、サイドバー高さ `height:100%` 修正。
- charts: **BarCell** プリミティブを移植（クロスタブ復元の前提部品）。

### Phase 2 — 欠落コントロール／状態の復元（FE中心）
- ~~booking-curve~~ **完了(2026-06-22)**: mart `booking_curve_lead_metrics` で売上/ADR/稼働率を実データ化。4指標マルチセレクタ＋すべて表示＋当年vs前年＋二軸4系列＋KPI撤去（案A実装）。
- occupancy: 比較モードセレクタ（前年実績/前年同期/指定日付＋date入力/予算差）＋ UnconfirmedPanel。
- channels: 当年/前年 Segmented トグル。
- occupancy: ActualMatrix に**予算 tfoot 行**を追加（コルディオ2025-26で実データ表示）。
- 追加KPI/ドーナツの整理（デザイン準拠で撤去/縮小）。

### Phase 3 — 中核クロスタブ表の復元（**要バックエンド対応**）
- **BE**: 次のグリッドを返す mart/API 契約を追加:
  - channels: 経路 × 施設（月間）／経路 × 12ヶ月（年間）
  - nationalities: 国籍 × 12ヶ月（指標別）
  - room-types: 部屋タイプ × 12ヶ月（指標別）
  - annual-sales: 12ヶ月 × 15施設（エリアグループ）
- **FE**: 各画面の中核クロスタブを再構築（多段スティッキーヘッダー、エリア/年colSpan、合計行＋合計列、ヒートシェーディング、構成比2段セクション）。
- nationalities/room-types: マルチ指標時の「指標ごとフルマトリクス積み重ね＋青いsectionBar」方式へ。
- analytics 各画面を `calc(100dvh-152px)` ノンスクロール・フィットへ回帰。

### Phase 4 — シェル仕上げ
- occupancy: AllFacilitiesOcc ビュー（AlertStrip＋4サマリーカード＋施設別比較テーブル：スティッキー先頭列・BarCell稼働率・前年差/予算差MetricDelta・状態Badge）。
- shell: UserMenu（アバター＋設定/サインアウト。ロールスイッチはスタブ）、DataFreshnessBadge（最終取込タイムスタンプ要）、ValidationBadge。
- shell/occupancy: モバイル対応（ドロワーサイドバー＋オーバーレイ、ボトムシートFilterSheet、モバイルサマリーバッジ列、occupancy モバイル Segmented）。

## 5. リスク／制約／前提

- **クロスタブ復元はバックエンド依存**。canonical 層に粒度（日付・施設・経路・国籍）が揃っているため**新規データ取得は不要**で、mart のグルーピング/API契約追加が主と見込む（Phase 3 冒頭で mart 粒度を要検証）。
- 予算・在庫データは**コルディオ2025-26のみ**存在。予算フッター/予算差/達成率はその範囲で実数、他は「—」。
- 国籍/部屋タイプの**月次クロスタブはAPI拡張前提**。現行の country×指標・roomType×指標は暫定。
- `.env.local`（鍵・DB接続・project ref）は**不可侵・ログ非出力**。
- プロトの内部構造（window グローバル/Babel/自前switch/`/1.1`税抜ハードコード）は**移植しない**（README方針）。視覚出力のみ再現。
